A production-minded Todo REST API built with FastAPI, SQLAlchemy, and JWT authentication.

This project demonstrates clean backend fundamentals for real-world apps: authentication, owner-based access control, layered architecture, filtering endpoints, test isolation, containerized deployment, PostgreSQL support, and soft delete.

## Project Overview

This API allows users to register/login, manage personal todos, and query time-based tasks (today/overdue). Each request is scoped to the authenticated user, and deleted items are preserved via soft delete (`deleted_at`) instead of hard deletion.

## Features

- JWT authentication (`register`, `login`, `me`)
- Owner isolation (users can access only their own todos)
- Todo CRUD endpoints
- Today and overdue todo filtering
- Soft delete with `deleted_at`
- SQLite support for local development
- PostgreSQL support with Docker Compose
- Pytest test suite with isolated test database
- OpenAPI docs via Swagger UI / ReDoc

## Tech Stack

- **Language:** Python 3.12
- **Framework:** FastAPI
- **ORM:** SQLAlchemy 2.x
- **Validation:** Pydantic
- **Auth:** JWT (`python-jose`) + password hashing (`passlib`)
- **Databases:** SQLite (local), PostgreSQL (Docker)
- **Testing:** Pytest + FastAPI TestClient
- **Containerization:** Docker + Docker Compose

## Architecture

Layered structure with clear responsibilities:

- **Router layer (`routers/`)**: HTTP endpoints, request/response handling
- **Service layer (`services/`)**: business logic orchestration
- **Repository layer (`repositories/`)**: data access and query logic
- **Model layer (`models/`)**: SQLAlchemy entities
- **Schema layer (`schemas/`)**: request/response contracts
- **Core/DB (`core/`, `db/`)**: security, dependencies, settings, DB session/base

Typical request flow:

```text
Client -> Router -> Service -> Repository -> Database
```

## Database Design

### `users`

- `id` (PK)
- `email` (unique)
- `hashed_password`
- `is_active`
- `created_at`

### `todos`

- `id` (PK)
- `title`
- `description`
- `is_done`
- `due_date`
- `created_at`
- `updated_at`
- `owner_id` (FK -> users.id)
- `deleted_at` (nullable, used for soft delete)

### Soft Delete Behavior

- `DELETE /api/v1/todos/{id}` sets `deleted_at = datetime.utcnow()`
- Todo queries only return records where `deleted_at IS NULL`
- Data is recoverable at DB level because rows are not physically removed

## Running Locally

### 1) Create virtual environment and install dependencies

```bash
python -m venv venv
# Windows PowerShell
.\venv\Scripts\Activate.ps1
# macOS/Linux
source venv/bin/activate

pip install -r requirements.txt
```

### 2) Run the API

```bash
uvicorn main:app --reload
```

App URLs:

- API: `http://127.0.0.1:8000`
- Health check: `http://127.0.0.1:8000/health`
- Swagger UI: `http://127.0.0.1:8000/docs`
- ReDoc: `http://127.0.0.1:8000/redoc`

Notes:

- Default local DB is SQLite via `sqlite:///./todo.db`
- You can override with `DATABASE_URL`

## Running with Docker

Build and run API + PostgreSQL:

```bash
docker compose up --build
```

Services:

- API: `http://localhost:8000`
- PostgreSQL: `localhost:5432`

Stop and remove containers:

```bash
docker compose down
```

If you need a fresh PostgreSQL volume:

```bash
docker compose down -v
docker compose up --build
```

## Running Tests

Run test suite:

```bash
pytest -q
```

Test setup highlights:

- Uses `tests/conftest.py`
- Creates isolated SQLite test database (`test_todo.db`)
- Overrides DB dependencies to avoid production/local DB pollution
